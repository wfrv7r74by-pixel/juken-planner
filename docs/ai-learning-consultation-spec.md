## 1. 目的

利用者から学習計画の生成に必要な情報を過不足なく収集し、
**「志望校合格」から逆算した週次タスク（教材名＋範囲まで具体化）** を継続的に提供する。

設計思想は大手学習塾の3つの型を統合したもの：

| 由来 | 取り込む要素 |
|---|---|
| 河合塾型 | 現在地と目標の距離を逆算した個別プラン設計 |
| 鬼管理型 | 週次PDCA面談（振り返り→要因分析→翌週設計）の定型化 |
| 武田塾型 | 教材名＋問題番号レベルまでのタスク具体化、4日2日1日の週次テンプレート、確認テストによる定着チェック |

### 非目標（やらないこと）

- 学習内容そのものの解説（質問対応は別機能。本機能は計画設計に限定）
- 志望校の合否判定・合格可能性の断定
- 5層すべてが埋まる前の週次タスク生成（**必ずブロックすること**）

## 2. 全体構造：ヒアリングの5層モデル

情報収集は以下の5層に分割し、**必ずこの順序**で進行する。

```
第1層 ゴール      … 志望校 / 入試日 / 受験方式 / 科目
第2層 現在地      … 模試データ（既存）or 代替指標＋診断テスト
第3層 総量        … 保有教材 / 進捗 / 学校の進度
第4層 可処分時間  … 部活 / バイト / 通学 / 平日休日の確保時間
第5層 個人特性    … 継続の癖 / 希望トーン / 報告頻度
```

**制約**：第1層・第2層・第4層が未取得の状態で週次計画を生成してはならない。
第3層・第5層は未取得でも仮値で進行可（後述のデフォルト値を使用）。

## 3. データモデル

`UserLearningProfile` を新規作成する。全項目に `confidence`（`confirmed` / `estimated` / `unknown`）を持たせ、推定値と確定値を区別できるようにすること。

```ts
type Confidence = 'confirmed' | 'estimated' | 'unknown';

interface Field<T> {
  value: T | null;
  confidence: Confidence;
  updatedAt: string;   // ISO8601
  source: 'user_input' | 'mock_exam' | 'diagnostic_test' | 'weekly_check' | 'default';
}

interface UserLearningProfile {
  userId: string;

  // ── 第1層：ゴール ──────────────────────────
  goal: {
    targetSchools: Field<Array<{
      name: string;
      faculty: string;
      priority: 1 | 2 | 3;
      examDate: string | null;      // ISO8601
      levelBand: 'top' | 'upper' | 'middle' | 'basic'; // 志望校未定時はここだけ埋める
    }>>;
    admissionType: Field<Array<'general' | 'common_test' | 'recommendation' | 'comprehensive'>>;
    subjects: Field<Array<{ code: string; status: 'fixed' | 'undecided' }>>;
    grade: Field<'hs1' | 'hs2' | 'hs3' | 'ronin'>;
    concurrentApplicationCount: Field<number>; // 併願想定数
  };

  // ── 第2層：現在地 ──────────────────────────
  currentLevel: {
    hasMockExam: boolean;
    mockExams: Field<MockExamResult[]>;        // 既存データを参照
    // 模試なしユーザーの代替指標
    proxyIndicators: {
      schoolName: Field<string>;
      schoolLevelBand: Field<'top' | 'upper' | 'middle' | 'basic'>;
      classRank: Field<{ rank: number; totalStudents: number }>;
      periodicTestScores: Field<Array<{ subject: string; score: number }>>;
      certifications: Field<Array<{ name: string; grade: string; score?: number }>>;
    };
    diagnosticResults: Field<DiagnosticResult[]>;
    unitMastery: Field<Array<{
      subject: string;
      unit: string;
      level: 0 | 1 | 2 | 3;   // 0:未習 1:未定着 2:基礎可 3:応用可
      verifiedBy: 'self_report' | 'diagnostic' | 'mock_exam' | 'weekly_check';
    }>>;
    // 学力不足と処理速度不足の分離用
    timeShortageRatio: Field<number>;  // 未回答率
  };

  // ── 第3層：総量 ────────────────────────────
  materials: Field<Array<{
    subject: string;
    title: string;
    totalUnits: number;         // ページ数 or 問題数
    completedUnits: number;
    laps: number;               // 周回数
  }>>;
  schoolProgress: Field<Array<{ subject: string; coveredUnits: string[] }>>;
  schoolAssignmentLoad: Field<'heavy' | 'normal' | 'light' | 'none'>;
  externalClasses: Field<Array<{ name: string; dayOfWeek: number; startAt: string; endAt: string }>>;

  // ── 第4層：可処分時間 ──────────────────────
  availability: {
    clubActivity: Field<{ active: boolean; retirementMonth: string | null; days: number[] }>;
    partTimeJob: Field<Array<{ dayOfWeek: number; startAt: string; endAt: string }>>;
    commuteMinutesPerDay: Field<number>;
    weekdayHours: Field<number>;      // 自己申告値
    weekendHours: Field<number>;      // 自己申告値
    fixedEvents: Field<Array<{ name: string; startDate: string; endDate: string; impact: 'block' | 'reduce' }>>;
    studyLocation: Field<Array<'home' | 'school' | 'library' | 'cram_school'>>;
  };

  // ── 第5層：個人特性 ────────────────────────
  traits: {
    pastFailurePattern: Field<string>;
    selfAssessedStrengths: Field<string[]>;
    selfAssessedWeaknesses: Field<string[]>;
    preferredTaskType: Field<'memorization' | 'practice' | 'no_preference'>;
    preferredTone: Field<'strict' | 'supportive'>;
    reportFrequency: Field<'daily' | 'weekly'>;
  };

  phase: 'onboarding' | 'diagnostic' | 'steady';  // 後述
  completeness: number;  // 0-100
}
```

## 4. 初回ヒアリングフロー

### 4-1. 絶対条件

- **初回の必須質問は10問以内**。それを超えると離脱する。
- 残りは週次相談の中で1回あたり1〜2問ずつ回収する（差分ヒアリング）。
- **一度取得した情報は二度と聞かない**。`UserLearningProfile` を必ず参照してから質問を生成すること。
- 回答形式は**選択式を基本**とする。自由記述は「過去に計画が崩れたきっかけ」など質的情報のみ。
- **全質問に「わからない／未定」の選択肢を設ける**。選択された場合は仮値を入れて先へ進む。聞き切ろうとして進行が止まるのが最悪の状態。

### 4-2. 初回必須10問（Must）

| # | 質問ID | 内容 | 形式 | 未回答時の仮値 |
|---|---|---|---|---|
| 1 | `goal.school` | 志望校・学部 | 検索＋選択 / 「未定」 | 未定→レベル帯のみ聞く（Q2へ） |
| 2 | `goal.levelBand` | （志望校未定時のみ）目標レベル帯 | 4択 | `middle` |
| 3 | `goal.admissionType` | 受験方式 | 複数選択 | `general` |
| 4 | `goal.subjects` | 受験科目 | 複数選択 / 「迷い中」 | 方式から自動推定 |
| 5 | `availability.club` | 部活の有無・引退予定 | 選択 | なし |
| 6 | `availability.job` | バイトの曜日 | 曜日複数選択 / なし | なし |
| 7 | `availability.hours` | 平日／休日に確保できる時間 | 各スライダー | 平日2h／休日5h |
| 8 | `materials.owned` | 持っている参考書（科目別） | 検索＋複数選択 / 「なし」 | なし |
| 9 | `level.entry` | 模試の有無 → 分岐（§5） | 分岐 | — |
| 10 | `traits.tone` | 希望するトーン | 2択 | `supportive` |

### 4-3. 週次で回収する項目（Should）

優先度順。1回の相談で最大2問まで。

1. 学校の授業進度（未習単元。数学・理科・社会で計画精度に直結）
2. 定期テスト日程・学校行事
3. 保有教材の進捗（何周目・どこまで）
4. 通学時間
5. 併願想定数
6. 過去に計画が崩れたきっかけ
7. 得意／苦手の自己認識
8. 学校課題の量
9. 主な勉強場所
10. 暗記系／演習系の得手不得手

## 5. 現在地の判定ロジック

### 5-1. 分岐

```
模試データあり  → ルートA（§5-2）
模試データなし  → ルートB（§5-3）
```

### 5-2. ルートA：模試データあり

既存の模試データから以下を自動抽出し、**ユーザーに再質問しない**。

| 抽出項目 | 用途 |
|---|---|
| 科目別偏差値・判定 | 科目間の時間配分比率の決定（苦手科目に厚く配分） |
| **単元別・設問別正答率** | 最重要。タスクの単元選定に直結。取得できていなければDBスキーマ拡張を提案すること |
| 模試の種類・実施時期 | 母集団補正（§5-4） |
| 前回からの推移 | 到達予測。生成した計画が現実的かの検算に使用 |

**模試データがあっても追加で聞く3問**（模試だけでは取れない情報）：

| 質問ID | 内容 | 用途 |
|---|---|---|
| `postmock.progress` | その模試以降に進めた学習内容 | 模試は過去の写像。直近のズレを補正 |
| `postmock.gap` | 手応えと結果がズレた科目 | 「わかったつもり」の検出 |
| `postmock.timeout` | 時間切れで解けなかった量 | 学力不足か処理速度不足かの分離。**処方が全く変わるため必須** |

### 5-3. ルートB：模試未受験

単一指標では精度が出ない。**以下5つを組み合わせて疑似的に精度を担保する。**

#### ① 母集団の分かる外部指標

- 学校の定期テストの点数 ＋ **学年順位／学年人数**
- **高校名（または高校の偏差値帯）**

この2つの掛け算で全国偏差値を推定する。「校内で真ん中」の意味は高校によって全く異なるため、**高校の学力帯情報を欠かすと推定が成立しない**。

- 英検／TOEIC等のスコア（英語は絶対評価で位置が測れる最も強い代替指標）
- 高1・高2の場合は高校入試時の得点・内申も可

#### ② アプリ内診断テスト（最優先で実装）

- 科目別 10〜15分、20問程度
- **易→難のグラデーション**をつけ、単元横断で構成
- 目的は点数ではなく**「どのレベルで躓くか」の特定**
  - 判定ロジック：あるレベルで **初見8割、または7割を2回** → 1つ上のレベルへ。躓いたレベルが現在地
- 制限時間を設け、**未回答／時間切れと誤答を必ず別カラムで記録**（処理速度の欠如か知識の欠如かを分離）

#### ③ 単元別チェックリスト

模試の設問別正答率の代用。ただし**自己申告は必ず甘く出る**。
→ **各単元に確認問題を1問添えて答え合わせさせる**こと。これだけで精度が大きく変わる。

例：英語＝関係詞／仮定法／分詞構文…、数学＝二次関数／確率／ベクトル…

#### ④ 保有教材からの出題

「持っている参考書の、すでに終えた範囲」から数問出題。学力測定と進捗確認を同時に処理できる。

#### ⑤ 運用でのカバー（最重要）

模試未受験者は**初期推定に誤差が残る前提**で組む。

- 初回計画は **`phase = 'diagnostic'`（最初の2週間）** として、**易しめ・少なめ**に設定する。過剰に盛ると即崩壊するため、**上方修正前提**で組むこと
- 毎週の確認テスト正答率を蓄積し、**2〜3週で推定レベルを自動補正**する。週次データが溜まれば模試1回分より高解像度になる
- **次回の全統模試・共通テスト模試の日程を提示し、申込をタスクとして計画に組み込む**。これは代替ではなく正攻法として必ず入れる

### 5-4. 母集団補正テーブル

模試種別ごとの偏差値換算表をコード内に定数として持たせる（進研模試は母集団が広く偏差値が高く出る等）。数値は実装時にユーザーへ確認すること。推測値をハードコードしないこと。

## 6. 計画生成ロジック

### 6-1. 生成順序

```
① 総量の可視化    ゴールまでに必要な教材・単元・問題数をリスト化
② 逆算            入試日までの残り日数で総量を除算 → 月間・週間の目安ペース
③ 個別調整        部活/バイト/学校行事/得意不得意で配分を補正
④ 週次テンプレ適用 §6-3 の型に流し込む
⑤ タスク具体化    教材名＋範囲（ページ／問題番号／単語番号）まで落とす
```

### 6-2. 可処分時間の算出

```
実効週間時間 = (平日申告値 × 5 + 休日申告値 × 2) × 0.8
             - 学校課題見込み
             - 塾・映像授業の固定枠
```

**自己申告値は必ず0.8掛けで見積もる。** 申告どおりに組むと確実に計画倒れになる。
通学時間は別枠として保持し、暗記系タスク（単語・古文単語・一問一答）専用の割当先とする。

### 6-3. 週次テンプレート（4日2日1日）

| 日 | 内容 |
|---|---|
| Day1〜4 | 新規内容を進める（インプット＋演習） |
| Day5〜6 | 直近4日間の復習 |
| Day7 | 確認日。定着度チェック＋翌週計画の調整 |

- 週の冒頭に**「今週のメインテーマ」を1つだけ**設定させる
- 曜日はユーザーの固定予定に合わせて割り当てる（バイトが月水金なら火木土に重い科目を配置）
- **Day7の出来をもとに翌週計画を生成する**こと。前週の結果を見ずに翌週を生成してはならない

### 6-4. タスクの粒度

**時間ベースではなく範囲ベースで出力する。**

- ✗ 「英語を2時間」
- ✓ 「システム英単語 401〜800（400語）を完璧に覚えきる」
- ✓ 「Next Stage 文法編 p.42〜58 の問題番号 210〜290」

必ず `教材名 + 開始位置 + 終了位置 + 目標到達度` の4要素を含めること。

### 6-5. 配分ルール

- 苦手科目（模試偏差値または診断結果で下位）に時間を厚く配分
- 得意科目は頻度を下げる（ゼロにはしない）
- **未習単元には演習タスクを割り当てない**（学校進度を必ず参照）
- 併願数に応じ、12月以降に過去問演習コマを事前確保する。**ここを確保しないと直前期が必ず破綻する**

## 7. 週次相談フロー（定型4ステップ）

毎週この4ステップを1セットとして実行する。

1. **前回設定したタスクの振り返り**（達成／未達をタスク単位で確認）
2. **未達の要因分析**（「なぜ」を1段階だけ掘り下げる。詰問にしない）
3. **確認テスト**：先週の内容から1〜2問出題
   - 正答 → 次へ進む
   - 誤答 → **次に進まず復習を優先**。「わかったつもり」を残さない
4. **翌週タスクの設計**（科目・教材・範囲まで具体化して提示）

加えて、模試結果・定期テスト結果の入力があれば方針修正に反映する。

### 7-1. 節目モード

以下のタイミングでは通常の週次相談と異なる専用テンプレートに切り替える。

| タイミング | モード内容 |
|---|---|
| 模試の直後 | 結果分析＋計画の大幅見直し |
| 出願期（10〜1月） | 併願校の確定支援＋過去問演習計画 |
| 共通テスト直前 | 総復習・時間配分演習中心へ切り替え |
| 部活引退月 | 学習時間の再算出＋計画のフェーズ移行 |
| 定期テスト2週間前 | 学習量を落とす前提でスケジュール再構成 |

## 8. 応答トーンの仕様

- ユーザーが選択した `preferredTone` に従う
- **未達を指摘する際は詰問調にしない。「原因分析 → 次の一手」に焦点を当てる**
- **達成できた部分は必ず明示的に評価する**（管理型サービスでも、承認があることが継続率に効く）
- 答えを直接与えるのではなく、考え方・解答過程を辿れる問いかけを優先する
- **合否の断定、他ユーザーとの比較、過度な不安の煽りは禁止**

## 9. 実装タスク分割

以下の順で実装すること。各ステップ完了時に動作確認を挟むこと。

1. `UserLearningProfile` の型定義とDBスキーマ／マイグレーション
2. 既存の模試データから第2層へのマッピング処理（単元別データの有無を先に調査）
3. 初回ヒアリングのUI（10問・選択式・「わからない」分岐つき）
4. 診断テストの出題エンジン（易→難、未回答と誤答の分離記録、躓きレベル判定）
5. 母集団補正・偏差値推定ロジック（ルートB用）
6. 可処分時間算出ロジック（0.8掛け、固定予定の控除）
7. 計画生成エンジン（総量算出→逆算→配分→4日2日1日への流し込み）
8. 週次相談の4ステップフロー
9. 差分ヒアリング（未取得項目を優先度順に1回1〜2問）
10. 節目モードの分岐

---

## 10. 受け入れ基準

- [ ] 初回ヒアリングが10問以内で完了し、全問に「わからない」の選択肢がある
- [ ] 模試データがあるユーザーに、模試から取得可能な情報を再質問しない
- [ ] 模試未受験ユーザーでも、診断テスト完了後に週次タスクが生成される
- [ ] 生成タスクが必ず `教材名＋範囲` を含み、「◯時間勉強」形式のタスクを含まない
- [ ] 第1・2・4層が未取得の状態では計画生成がブロックされ、不足項目が提示される
- [ ] 部活・バイト・学校行事の時間帯にタスクが割り当てられない
- [ ] 未習単元に演習タスクが割り当てられない
- [ ] 週次相談が4ステップ順で実行され、確認テスト誤答時に復習へ分岐する
- [ ] 2週間分の週次データ蓄積後、推定学力が自動補正される
- [ ] 一度回答済みの質問が再提示されない

### テストケース例

| ケース | 期待挙動 |
|---|---|
| 志望校未定・模試なし・高2 | レベル帯仮置き＋診断テスト実施、`phase='diagnostic'` で易しめの計画を生成 |
| 部活週6・平日1h申告 | 実効時間 = (1×5+休日値×2)×0.8。引退月で計画がフェーズ分割される |
| 確認テスト誤答 | 翌週計画で新規範囲を進めず、該当単元の復習が優先配置される |
| 定期テスト2週間前 | 学習量が自動的に減り、学校科目の比重が上がる |
| 数学の未習単元を含む志望校 | 学校進度到達までその単元の演習タスクが生成されない |

## 11. 実装上の禁止事項

- 5層が揃わない状態で週次タスクを生成すること
- 推定値を確定値として扱うこと（必ず `confidence` を保持・表示）
- 自己申告の可処分時間をそのまま計画に反映すること
- 時間ベースのタスクを出力すること
- 同じ質問を再提示すること
- 模試種別の換算値など、確認していない数値をハードコードすること
